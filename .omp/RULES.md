# EzWinCommand 项目开发规范

## 多 Agent 开发流程

每次功能开发必须启用完整的 **SE → Dev → Test** 流程，禁止跳过：

| 角色 | 职责 | 输出 |
|---|---|---|
| **SE**（需求分析师） | 拆解需求、方案对比、输出设计文档 | `local://se-design-*.md` |
| **Dev**（开发工程师） | 按设计文档实现，各模块并行开发 | 代码 + commit |
| **Test**（测试工程师） | 设计用例、发掘 bug、提交修复 | 测试用例 + bug 报告 |

- 禁止不经过 SE 分析直接写代码。
- 禁止不经过测试就声称完成。

## Git 提交纪律

- 每次编辑文件实现代码后 **立即** `git commit`。
- 每次修复 bug 后 **立即** `git commit`。
- 提交格式：`feat:` / `fix:` / `refactor:` / `chore:` 前缀。
- 禁止多个独立改动合并成一个 commit。
- 保持提交粒度细、描述清晰。

## 设计原则

- **零额外依赖优先** — 能用标准库或已有依赖解决的，不加新包。
- **Windows 原生 > 第三方** — 同一功能优先 `win32api` 而非第三方封装。
- **方案对比量化** — 每个设计决策必须列出代码量、依赖增量、磁盘占用、复杂度等维度的对比表。

## 技术栈

- Python 3.13，Windows 11
- FastAPI + Uvicorn
- pywin32（已有依赖，零增量）
- 纯静态 HTML+JS Web UI
