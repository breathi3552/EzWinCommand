# 产品与测试资产维护

长期事实写入 `docs/product/`，测试资产索引写入 `docs/tests/`，版本摘要写入 `docs/versions/`。不要为每轮任务创建需求目录或复制测试日志。

## 产品地图

每个模块记录：

- 职责与边界。
- 特性和功能点。
- 用户路径及重要失败/恢复行为。
- 协议、安全和平台约束。
- 对应长期测试用例 ID。
- 尚未自动化、需要真实环境或待人工验证的路径。

改变用户可观察行为、协议或模块边界时，由 Main 在候选交付进入验证前更新。Bug 仅恢复既有行为时，通常只更新测试覆盖和版本日志。

## 长期测试资产

- 使用稳定用例 ID，格式 `TC-{AREA}-{NNN}`。
- 功能迭代时修改或废弃旧用例，不把过期断言永久叠加。
- `docs/tests/coverage-map.md` 维护“模块 → 特性 → 功能点 → 用例 → 自动化层级 → 实现位置”。
- Test Verifier 只报告漂移或缺口；Main 负责修改文档和自动化脚本。
- 代码覆盖率记录为诊断信息，业务功能覆盖才是门禁。

## 版本日志

在 `docs/versions/v{major}.{minor}.md` 追加简短条目：日期、用户可见变化、修复、协议/安全变化、待人工验证项。避免记录内部过程流水账。
