# 验证结论协议

## Test Verifier

```text
结论：通过 | 通过（待人工验证） | 失败 | 阻塞
业务覆盖：完整 | 不完整 | 未知
自动化检查：...
AI 环境检查：...
人工验证：无 | 待人工验证：...
失败项：无 | ...
证据：...
返回对象：目标审查 | Main | 用户
```

结论约束：

- “通过”要求业务覆盖完整，所有计划内检查均已由自动化或 AI 环境测试完成，人工验证为“无”。
- “通过（待人工验证）”同样要求业务覆盖完整；所有未自动化项目都必须逐项列入明确的人工覆盖范围，不得存在“未知”或无归属缺口。
- 业务覆盖为“不完整”时必须返回“失败”；业务覆盖为“未知”时必须返回“阻塞”。
- 涉及安全、鉴权、数据破坏或不可恢复副作用的关键人工项未完成时必须返回“阻塞”。

## Goal Reviewer

```text
结论：通过 | 失败 | 阻塞
目标覆盖：G-1=已覆盖 | 缺失 | 不明确；...
验收覆盖：AC-1=已覆盖 | 缺失 | 不明确；...
无目标映射的变更：无 | ...
遗漏范围：无 | ...
非目标影响：无 | ...
产品资料漂移：无 | ...
证据：...
返回对象：Release | Main | 用户
```

## 汇总

- Test“通过”且 Goal“通过” => 总结论“通过”。
- Test“通过（待人工验证）”且 Goal“通过” => 总结论“通过（待人工验证）”。
- 任一“失败” => Main 修复后重验。
- 任一“阻塞” => 向用户报告阻塞。
- 同一失败连续三轮出现 => 停止循环并报告用户。
- 缺少任一规定字段或字段值互相矛盾 => 不得放行，返回“阻塞”。
