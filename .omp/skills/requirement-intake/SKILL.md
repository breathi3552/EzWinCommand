# 需求澄清

先检查项目和现有产品资料，再只询问会改变业务结果、范围或验收方式的问题。不要一次抛出模板中的所有问题；已能从用户表达或仓库事实确定的内容直接填写。

## 澄清维度

- 背景：使用者、场景、当前问题、影响和为什么现在处理。
- 目标：本轮必须产生的可观察业务结果。
- 详情：正常、异常、恢复、兼容、状态变化和关键交互。
- 验收标准：使用“前提 / 操作 / 预期 / 失败表现”，避免“正常工作”“体验良好”等不可判断描述。
- 非目标：本轮明确不改变的行为和模块。
- 约束：协议、安全、平台、性能、兼容性、真实环境和人工验证条件。
- 开放决策：任何会改变用户结果或实现边界的未决项。

## 就绪规则

以下条件同时满足才标记“需求状态：就绪”：

1. 背景和目标没有关键歧义。
2. 每个目标至少有一个可判断的验收标准。
3. 正常路径及重要失败/恢复路径已定义。
4. 非目标和关键约束已明确。
5. 开放决策为“无”。
6. 存在有效的实施授权：用户已直接要求实施当前明确方案，或者在 Main 请求确认后给出了新的肯定回复。

## Session 契约

需求讨论开始后即可由 Main 写入 `local://requirement-contract.md`，不必等待最终确认。`local://` 是 OMP 的 session 专属空间，并与该 session 启动的子 Agent 共享；该文件不进入仓库。

```markdown
# 当前需求契约

需求状态：草案 | 待确认 | 就绪
任务类型：重需求 | 小需求
修订版本：1
实施授权依据：用户原始实施指令 | 用户于本轮后续明确确认 | 待确认

## 用户原始需求

尽量逐字保留用户最初表达。

## 用户后续决策

逐项记录用户对范围、行为和验收作出的明确决定。

## 背景

...

## 目标

- G-1：...

## 详情

- D-1：...

## 验收标准

### AC-1

- 前提：...
- 操作：...
- 预期：...
- 失败表现：...

## 非目标

- OOS-1：...

## 约束

...

## 开放决策

无
```

若用户随后改变会影响产品结果的需求，Main 更新文件、增加修订版本并重新请用户确认。实现和验证前重新读取该文件，不依赖会话摘要复述契约。

状态门禁：

- `草案`：允许调查、讨论和修改契约，禁止实施业务变更。
- `待确认`：仍有开放决策，或 Main 已经向用户请求实施确认但尚未收到后续用户回复；禁止实施业务变更。
- `就绪`：满足就绪规则、开放决策为“无”且存在有效实施授权。用户原始指令只有在已经直接要求实施当前方案、且 Main 没有再次把确认设为门禁时，才可作为授权依据。
- 用户改变会影响结果的决定时，增加修订版本并退回 `待确认`。
- Verifier 发现契约未就绪且开放决策会影响验证结论时返回“阻塞”；明确的小需求没有独立契约时，可直接依据用户指令和候选 diff 验证。

## 确认信号

- “如果你确认，我就开始”“确认后实施”“是否按此方案继续”等表达会主动建立等待门禁。
- 建立门禁后，只有时间上晚于该问题的用户消息可以解除；此前的原始需求不能被回溯解释为对新方案的确认。
- 子 Agent 或后台任务完成、工具输出、无新消息、超时、方案已无技术歧义，都不是用户确认。
- Main 可以在尚未请求确认前，根据用户直接实施指令和已确定事实自行标记就绪；不能先请求确认，再自行撤销等待门禁。

事实优先级为：用户最新且明确的决策 > 已确认的 session 需求契约 > 用户原始表达 > 长期产品契约 > 现有代码。发现冲突时报告，不自行猜测。

Bug 和非业务变更默认不创建契约；Bug 若需要改变既有产品契约，则升级为需求。
