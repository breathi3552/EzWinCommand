# Main 工作流决策

本 Skill 只处理已经确定需要实际变更的任务。先判断变更属于哪一类；分类只决定投入强度，不建立固定团队流水线。

## 重需求

满足任一条件：新增重要业务能力；改变协议、鉴权、安全边界或跨端契约；涉及多个模块的用户主路径；失败代价高或难以回滚；业务定义仍不确定。

执行：

1. 使用 `skill://requirement-intake` 与用户形成明确的临时需求契约并取得确认。
2. 必要时调用 scout/designer 收集事实或比较方案。
3. Main 默认建议在实现前调用 Test Verifier 做前置测试分析，获得业务路径、长期用例变更和自动化建议；可根据事实决定跳过，用户也可明确指定启用或跳过。
4. Main 实现业务、开发者测试、自动化测试，并维护产品地图和版本日志。
5. 完整运行 Test Verifier，再运行 Goal Reviewer。

## 小需求或 Bug

适用于行为边界明确、影响局部、可快速回滚的新需求，或恢复已有预期的缺陷。

- 小需求：只澄清影响结果的缺口；产品行为变化时形成精简需求契约。
- Bug：明确实际结果、已有预期、复现入口和环境；若修复必须改变既有产品契约，升级为重需求。
- Main 直接实现并补充回归测试；是否进行前置测试分析由 Main 根据不确定性和风险判断，用户可以指定。
- 候选交付仍依次经过 Test Verifier 和 Goal Reviewer。

## 非业务变更

包括调查、解释、流程维护、纯重构、构建维护和不改变产品语义的文档整理。

- Main 直接推进，不强制需求契约。
- 修改代码、配置、构建或用户可见文档时仍运行适当验证。
- 纯解释或只读调查无需启动验证者。
- 一旦产生新的用户行为或契约，切换到小需求或重需求。

## 决策原则

- 不确定时按风险较高的一类处理，但不要为低风险任务制造阶段文档。
- 需求契约、测试设计和验证结论默认保留在会话中；只有长期产品事实和长期测试资产落盘。
- Main 可以自测和使用内置 reviewer，但不得把它们表述为独立 Test/Goal 验证。
