# Main 工作流决策

本 Skill 只处理已经确定需要实际变更的任务。先判断变更属于哪一类；分类只决定投入强度，不建立固定团队流水线。

## 重需求

满足任一条件：新增重要业务能力；改变协议、鉴权、安全边界或跨端契约；涉及多个模块的用户主路径；失败代价高或难以回滚；业务定义仍不确定。

执行：

1. 先依据仓库事实确认产品影响，再使用 `skill://requirement-intake` 形成 session 需求契约；存在会改变结果的开放决策时，不实施受该决策影响的业务变更。
2. 必要时调用 scout/designer 收集事实或比较方案。
3. Main 默认建议在实现前调用 Test Verifier 做前置测试分析，获得业务路径、验收覆盖变化和自动化建议；可根据事实决定跳过，用户也可明确指定启用或跳过。
4. Main 实现业务并完成开发者测试与实际 smoke。
5. Main 在独立验证前维护确有长期价值的产品契约、协议契约、验收覆盖和变更日志。
6. 完整运行 Test Verifier，再运行 Goal Reviewer。

## 小需求或 Bug

适用于行为边界明确、影响局部、可快速回滚的新需求，或恢复已有预期的缺陷。

- 小需求：只澄清影响结果的缺口；用户最初已明确授权实施且 Main 未另行请求确认时，可直接形成就绪的精简契约，不重复请求形式确认。Main 一旦明确请求确认，就必须等待后续用户回复。
- Bug：明确实际结果、已有预期、复现入口和环境；若修复必须改变既有产品契约，升级为重需求。
- Main 直接实现并补充回归测试；是否进行前置测试分析由 Main 根据不确定性和风险判断，用户可以指定。
- Main 完成针对性自测；存在跨模块影响、目标歧义或测试覆盖不确定时选择相应 Reviewer，局部且证据明确的改动无需固定启动两个 Reviewer。

## 非业务变更

包括调查、解释、流程维护、纯重构、构建维护和不改变产品语义的文档整理。

- Main 直接推进，不强制需求契约。
- 修改代码、配置、构建或用户可见文档时仍运行适当验证。
- 纯解释或只读调查无需启动验证者。
- 一旦产生新的用户行为或契约，切换到小需求或重需求。

## 决策原则

- 多步骤开发任务的计划默认只保留 3–7 个真实阶段；单步修改不创建计划。不得在搜索仓库前编造路径、调用方、跨端影响或测试结果。
- 只显式记录会改变范围、成本、验证强度或阶段权限的决定；普通问答、明确 Bug 和显然不适用的测试层级无需完整摘要。
- Main 根据可回滚性、影响范围、证据质量和失败代价选择投入强度；不确定性是调查信号，不自动升级为最重流程。
- 需求契约、测试设计和验证结论默认保留在会话中；只有长期产品事实和长期测试资产落盘。
- Main 可以自测和使用内置 reviewer，但不得把它们表述为独立 Test/Goal 验证。

## 阶段决策摘要

只在决定偏离默认路径、存在开放风险或需要用户知情时输出摘要；默认项不逐项填写。摘要用于监督，不是审批表。

```text
决策摘要
- 任务分类：重需求 | 小需求 | Bug | 非业务变更
- Git 边界：干净 | 本轮路径与现有改动不重叠 | 用户已确认纳入
- 需求状态：不适用 | 草案 | 待确认 | 就绪
- 前置测试分析：启用 | 跳过；原因：...
- 长期资料：更新 ... | 不更新；原因：...
- UI/E2E：需要 | 不需要；AI/人工验证：无 | ...
- 当前门禁：允许调查 | 等待确认 | 允许实施 | 准备独立验证
```

按需交付顺序为：调查与必要契约 → Main 实现 → 针对性自测/smoke → 必要长期资料 → 按风险选择 Test Verifier、Goal Reviewer → 用户要求 Git 操作时进入 Release。验证后若候选交付变化，只重新检查受影响内容。
