# E2E 与 AI 环境测试

只在 Test Verifier 后置验证或 Main 调试时按需读取。验证者保持只读；若需要新建或修改脚本，输出缺口并交给 Main。

## 优先主链

1. 启动隔离测试配置的真实 Server，确认 `/ping`。
2. 启动 Android 模拟器，使用 ADB 安装 Debug APK。
3. 通过 UI 完成地址输入、配对、进入控制页和核心控制路径。
4. 同时验证 Web 本机管理端的配对码生成和设备管理。
5. 保存命令、关键 UI 状态和服务端结果之间可追溯的证据。

## 自动化层级

优先使用稳定语义定位和确定性断言。新需求尚无脚本、业务状态不确定或结果依赖真实 Windows UI 时，可以由 AI 操作环境、观察结果并记录步骤与证据。只有稳定且重复执行的路径才沉淀为长期脚本。

## 证据要求

- 环境：OS、Android API/设备、Server 地址和构建标识；敏感值脱敏。
- 步骤：用户可复现的输入与动作。
- 预期与实际：分别陈述。
- 证据：测试输出、HTTP 结果、UI 截图或结构化 UI dump、必要日志。
- 结论：Automated、AI-assisted 或 Manual，三者不得混写。

## 人工验证

未执行写“待人工验证”。关键安全、鉴权、数据破坏或不可恢复副作用的人工项未完成时返回“阻塞”；其他路径在全部未自动化项目都有明确人工覆盖范围时可返回“通过（待人工验证）”。
